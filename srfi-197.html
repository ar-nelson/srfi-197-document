Abstract

Many functional languages provide chaining or pipeline operators, like Clojure's -> or OCaml's |>. Chaining is a simple, terse, and readable way to write deeply nested expressions. This SRFI defines a family of chain operators, which can rewrite nested expressions like (a b (c d (e f g))) as a sequence of operations: (chain g (e f) (c d) (a b)).

Rationale

Deeply nested expressions are a common problem in all functional languages, especially Lisps. Excessive nesting can result in deep indentation and parenthesis-matching errors.

content_copy
; Quick, how many close parentheses are there?
(eta (zeta (epsilon (delta (gamma (beta (alpha)))))))

Additionally, some expressions sound more natural when written inside-out, as a sequence of steps from start to finish.

content_copy
; This recipe looks… backwards.
(bake (fahrenheit 350) (pour (mix (add eggs (add sugar (add flour bowl))))))

Many functional languages solve this by introducing expression chaining operators (sometimes called "pipeline" operators). This SRFI defines a chain operator inspired by Clojure's threading macros, particularly the ->> operator.

content_copy
(chain (alpha) (beta) (gamma) (delta) (epsilon) (zeta) (eta))

(chain bowl
       (add flour)
       (add sugar)
       (add eggs)
       (mix)
       (pour)
       (bake (fahrenheit 350)))

Expression chaining is especially useful for nested list and vector operations, in which it is often difficult to match the last argument (the list or vector) to the correct procedure call because they may be separated by a multi-line lambda.

content_copy
(chain xs
       (map (lambda (x) (+ x 1)))
       (filter odd?)
       (fold * 1))

Scheme already provides an idiomatic way to chain expressions in let* and SRFI 2 and-let*, but the primary advantage of chain is terseness and the accompanying readability. This focus on terseness and reduced nesting is similar in spirit to SRFI 26.

Compared to an equivalent let* expression, chain removes one level of parenthesis nesting, does not define any intermediate variables, and may omit the final argument of each step. It also allows mixing single and multiple return values.

To demonstrate the difference in verbosity, here is the let* equivalent of the recipe expression:

content_copy
(let* ((x bowl)
       (x (add flour x))
       (x (add sugar x))
       (x (add eggs x))
       (x (mix x))
       (x (pour x)))
  (bake (fahrenheit 350) x))

Specification

chain

Takes an initial pipeline value followed by any number of steps. Each step is a list that is evaluated as a call, with the pipeline value inserted as its final argument. The steps are evaluated in order, with the return value of each step becoming the pipeline value of the next step. The return value of chain is the return value of the last step.

content_copy
(chain x (a b)) ; => (a b x)
(chain (a b) (c d) (e f)) ; => (e f (c d (a b)))

The pipeline value may be moved to a different location in the step, using the keyword <>. This syntax is modeled after SRFI 26. <> may only appear in the top-level form of a step, not in any nested forms.

content_copy
(chain (a b) (c <> d) (e f)) ; => (e f (c (a b) d))

Multiple pipeline values are supported. If a step returns multiple values, the following step must contain as many <> arguments as the number of pipeline values. The values will be passed in left-to-right order.

content_copy
(chain (a) (b <> <>) (c))
; => (let-values (((<>1 <>2) (a))) (c (b <>1 <>2)))

SRFI 26's <...> syntax is also supported. <...> receives all remaining pipeline values and expands them into the step as multiple arguments.

content_copy
(chain (a) (b <> c <...>) (d))
; => (let-values (((<>1 . <>2) (a))) (d (apply b <>1 c <>2)))

Keep in mind that steps that do not contain <> always expect one pipeline value. That is, the step (a) is equivalent to (a <>), not (a <...>). Multiple values must always be explicitly specified.

chain-and

A variant of chain that short-circuits and returns #f if any step returns #f. chain-and is to chain as SRFI 2 and-let* is to let*.

Because chain-and checks the return value of each step, it does not support steps with multiple return values. Each step may contain no more than one <> argument, and cannot contain <...>.

chain-when

A variant of chain in which each step has a guard expression. Each argument after the first is a clause (<guard> <step>), where <step> is a step with the same syntax as a chain step. Clauses with no guard may be written as one-element lists (<call>), as in SRFI 2. Clauses whose <guard> returns #f will be skipped.

content_copy
(define (describe-number n)
  (chain-when '()
    ((odd? n) (cons "odd"))
    ((even? n) (cons "even"))
    ((zero? n) (cons "zero"))
    ((positive? n) (cons "positive"))))

(describe-number 3) ; => '("positive" "odd")
(describe-number 4) ; => '("positive" "even")

Because chain-when may skip steps, it does not support steps with multiple return values. Each step may contain no more than one <> argument, and cannot contain <...>.

chain-lambda

Creates a procedure from a sequence of chain steps. The procedure's arguments become the initial pipeline values.

content_copy
(chain-lambda (a) (b)) ; => (lambda (<>) (b (a <>)))

If the first step contains multiple <> arguments, the procedure takes multiple arguments. If the first step contains <...>, the procedure is variadic.

content_copy
(chain-lambda (a <> <>) (b c)) ; => (lambda (<>1 <>2) (b c (a <>1 <>2)))

Implementation

A sample implementation is available on GitHub. This implementation uses only syntax-rules macros, and is written in portable R7RS-small. Its only dependency is SRFI 2. It includes an R7RS library wrapper and a test script.
